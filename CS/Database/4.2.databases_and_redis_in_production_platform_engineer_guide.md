## 3. Relational Database (Postgres) – Production Setup

### 3.1 Primary + Replica Architecture

```
Application
    |
    | writes
    v
Primary Postgres  ---->  Replica Postgres
```

- Primary handles **writes**
- Replica continuously copies data
- Replica is used for:
  - Failover
  - Read scaling (non‑critical)

---

### 3.2 Why Replica Exists

- Not for backup
- For **high availability**

If primary fails:
- Replica promoted
- Application reconnects

---

## 4. Redis Cache – Production Architecture

### 4.1 Redis Placement

Redis always sits **between application and database**.

```
User → App → Redis → App → User
              ↓ (miss)
           Database
```

---

### 4.2 Redis Primary + Replica

```
App → Redis Primary → Redis Replica
```

- Primary handles writes
- Replicas handle reads
- If primary fails → replica promoted

Redis replicas are for:
- Availability
- Read scaling

Redis replicas are **NOT backups**.

---

## 5. HPA and Redis – Important Truth

### Do we use HPA for Redis?

❌ **Not for scaling data**

HPA:
- Scales stateless workloads
- Cannot split in‑memory data

### What we do instead

- Fixed replicas (hard‑coded)
- Redis Cluster (sharding) for scale
- Add replicas for HA, not HPA

👉 **Redis scaling is intentional, not automatic**.

---

## 6. Kubernetes Deployment Patterns

### 6.1 Why Redis Needs Dedicated Nodes

Redis is **memory‑intensive**.

So we use:
- Dedicated node pool
- High RAM nodes
- Taints + tolerations

This prevents:
- App pods stealing memory
- OOMKills

---

### 6.2 Node Selection (Conceptual)

```
[ Memory Node Pool ]  ← Redis only
[ App Node Pool ]     ← Applications
```

Redis pods:
- NodeSelector → memory nodes
- Resource limits set

---

## 7. Failure Scenarios (Real Production)

### 7.1 Database Primary Down

What happens:
- Primary crashes
- Replica promoted
- App reconnects via DNS

Platform responsibility:
- Automatic failover
- Stable endpoint
- Retry logic

---

### 7.2 Redis Eviction

Cause:
- Memory full

Effect:
- Cache miss
- DB hit
- Possible DB overload

Prevention:
- TTL on all keys
- Proper sizing
- Dedicated nodes
- Monitoring eviction rate

---

### 7.3 Replica Lag

Replica lag means:
- Replica has old data

Danger:
- Wrong reads
- Stale cache

Mitigation:
- Critical reads → primary
- Cache filled from primary
- Monitor lag

---

## 8. High Availability Concepts

### Active‑Passive

```
Active DB
   ↓
Passive DB
```

- Simple
- Strong consistency

---

### Active‑Active

```
App → DB‑1
App → DB‑2
```

- Used for caches
- Eventual consistency
- Complex conflict resolution

---

## 9. Backups, RPO, RTO (VERY IMPORTANT)

### Backups

- Automated
- Stored off‑cluster
- Tested restores

---

### RPO (Recovery Point Objective)

> How much data loss is acceptable

Example:
- RPO = 5 minutes
- Lose max 5 minutes of data

---

### RTO (Recovery Time Objective)

> How fast system must recover

Example:
- RTO = 10 minutes
- System must be usable in 10 mins

---

## 10. How Applications Connect (Networking)

### Kubernetes Service + DNS

```
App → Service → DB / Redis Pod
```

Why:
- Pods change IPs
- Services are stable

Apps never connect to pod IPs.

## 13. Interview‑Style Questions & Answers

### Q: Why not use Redis as DB?
A: Redis is volatile and can evict data. DB is source of truth.

---

### Q: Do you use HPA for Redis?
A: No. Redis scaling requires sharding or replicas, not HPA.

---

### Q: What happens if Redis is down?
A: System degrades gracefully and reads from DB with protection.

---

### Q: Replica vs Backup?
A: Replica = availability. Backup = recovery.
